## M1-W1-1 (2026-07-15) · ReAct 精讲

### 错误 #1: Observation 与 Thought 混淆

**触发场景**: 首次讲解 ReAct 三元组后的费曼自测

**我说错的话**:
> "Observation 就是拿到 tools 返回的数据后，总结看是否已经能回答提出的问题的过程"

**真相**:
- Observation = 工具返回的**原始数据** (被动接受)
- Thought = LLM **主动推理**的输出

**为什么会混**: 直觉上"看结果"包含"判断结果", 但工程实现里这是**两个独立阶段**

**修复方式**: 用广告归因类比强化
```
Action:      call service("get uin by device_id")
Result:      "uin=12345"     ← 服务返回
Observation: "uin=12345"     ← 【就是】result, 不加工
Thought:     "拿到 uin 了, 下一步查点击数据"  ← 这才是判断
```

**元原则**: **数据的接收 与 数据的解读 是两回事** —— 这个错误在很多地方都会犯

---

### 错误 #2: 认为 result 需要 LLM 加工才是 Observation

**触发场景**: 边界练习 (伪代码找 bug)

**我说错的话**:
> "step4 错在把 result 直接当 observation, 应该要多一步将 result 继续 think, think 的结果才是 observation"

**方向反了**: 我加了一步 (应该去掉), 而不是去掉一步 (原代码多了一步)

**真相**: `observation = result`, 一字不改, 直接塞 context

**3 个死因**:
1. **信息损失** - 加工时字段被总结掉
2. **幻觉入口** - 多 1 次 LLM 调用 = 多 1 次瞎编
3. **成本翻倍** - 4 轮从 4 次调用变 8 次

**深层原因**: 破坏 **Faithfulness (忠实性)** → 错误指数级放大

**修复方式**: 反过来用自己的类比论证
```
广告归因场景:
  服务返回 "uin=12345"
  ↓
  不会叫"归因专家"来解读这个 uin
  ↓
  直接塞给下一步用就完了
  
ReAct 也一样, Observation 就是 raw result
```

**元原则**: **不要过度设计, 相信下游的能力** ——
LLM 就是用来读原始数据的, 不需要"预处理管家"

---

## 🧬 元错误分析: 我为什么会犯这两个错?

```
共同根源:
  「工程直觉里, 数据流转必须经过'加工/校验/翻译'层」
  
     ↑ 这个直觉在传统系统里是对的 (微服务/ETL/API 网关)
     
  但在 LLM Agent 里, LLM 本身就是那个"加工层"
     ↓
  再加一层 LLM 加工 = 冗余 + 有害
```

**未来警戒**: 每次想在 LLM 和 context 之间加"预处理"时, 停下问一句:
> "这个预处理, LLM 自己下一轮是不是就会做?"
> 如果是, 就别加。

## 🎯 修正后的正确心智模型

```
❌ 我原来的模型:
   [tool] → [result] → [LLM 加工] → [Observation] → [LLM 判断] → [Thought]
                          ↑ 冗余!
   
✅ 正确的模型:
   [tool] → [result = Observation] → [塞 context] → [下轮 LLM: Thought]
                                                       ↑
                                              一次 LLM 调用同时产出
                                              解读上一轮 + 决定下一轮
```

## 📚 术语档案（本次首次接触）

- **Faithfulness (忠实性)**: Observation 必须忠实反映外部真实状态
- **Type Drift (类型漂移)**: LLM 加工数据导致的类型/语义偏移 (⭐ 我独立推导出的概念)
- **Agent 幻觉复合**: 失真数据在循环中指数级放大
