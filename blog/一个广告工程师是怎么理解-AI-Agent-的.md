## 1. Agent 是什么

Agent = LLM + 工具 + 自主决定"下一步做什么"。

**展开来说**，Agent是基于LLM之上，把提示词，工具调用，和事实结果，通过推理范式或者叫认知架构的方式，整合起来，最终达到自主决策，进而完成人类的任务目的的这么一套体系，或者叫机制。

至于传统的程序服务，在一定程度上也是为了帮助人类完成既定的任务，这里的区别主要是在泛化能力上，传统的程序服务，只能在既定的逻辑范围内运行，通过if/else针对预期内的条件做分支，处理不同类型的任务，但是Agent可以在这个基础上，由LLM根据实际情况自主决定走哪个分支，不局限于某些既定的任务，可以自主的完成更多的任务，不再需要机械的通过if/else来控制。

比如在广告系统中，要计算一个用户的归因数据，要分别拉取用户的状态数据，广告的点击数据，归因的配置数据，然后再进行逻辑计算，传统的系统服务就是用代码写具体的逻辑，通过接口调用或者函数调用完成用户数据，点击数据和配置数据的拉取；在Agent里这些数据的拉取都是tool调用，然后由LLM决定调用哪些tool，来最终完成归因结果的计算。

## 2. 什么时候用 Agent，什么时候不用

如果当前的业务场景，可以通过条件来穷举，且逻辑清晰通顺，那么更适合使用workflow。条件能穷举时就该老老实实写if/else更稳定，更准确，成本更低。

反之，则可以考虑使用Agent来解决问题，具体来说是条件无法一一列举，就算目前能列举，后期也不好维护的场景。

比如要诊断广告报表延迟了的告警，可能需要评估的条件很多，媒体侧api，自身服务资源，最近是否做过线上发布，配置修改等。

除此之外后期可能还会引入别的因素，比如周环比，月环比等，是无法穷举的，就是用走Agent的道路。

一句话：Agent 是用灵活性和成本，去换确定性覆盖不到的那片灰色地带。

## 3. Agent 怎么想

那么现在决定使用Agent了，就得知道Agent的思考路径，能帮我们更好理解他的工作机制，同时与我们的思考方式达成共识，更好地完成任务。

总的来说，分成三种模式CoT，ReAct和Plan。具体来说是：

1. CoT，Chain of Thought，思维链，通过大模型的历史训练资料思考，推理，最终得到答案，通常用于回答比较固定的知识，如数学题，逻辑推理。

2. ReAct，Reasoning + Acting，推理和行动，通过工具调用，结合实际情况以边做边思考的方式，最终得到答案。

3. Plan-and-Execute，计划再行动，在做任务之前先做计划，将任务拆为具体哪些步骤，计划完成后再完全按照计划实际实施。

以之前提到的诊断广告报表延迟case为例，3种模式会如何执行

1. CoT：根据历史训练的经验数据，可能会告知当前服务器负载高，可能会告知媒体侧有异常，可能会告知这是正常现象。

2. ReAct：先去捞当前服务器负载，发现没问题，再去抽样媒体API数据看返回是否延迟等一系列动作确认后，最后得出结论。

3. Plan-and-Execute：先规划要查 a.服务器负载数据;b.抽样媒体侧API数据;c.版本变更数据，得到数据后看哪部分有问题，然后得到结论。

| 模式 | 什么时候用 | 一句话判断 |
|------|-----------|-----------|
| **CoT** | 纯推理，不需要查外部数据 | 「这道题我脑子里能算出来」 |
| **ReAct** | 需要实时数据，步骤间有依赖，不确定性强 | 「我先查一下，根据结果再决定下一步」 |
| **Plan-and-Execute** | 任务明确可拆解，结果可预期 | 「我先列个清单，按清单一步步来」 |

## 4. Agent 怎么做

```
User 提问
    │
    ▼
┌──────────┐     ┌──────────┐     ┌──────────┐
│  LLM 想  │ ──► │  调工具  │ ──► │  看结果  │
└──────────┘     └──────────┘     └──────────┘
       ▲                                │
       └──────────── 循环 ◄─────────────┘
                        直到 LLM 不再调工具
```

知道了Agent怎么想的，也要知道Agent是怎么做的。

如上图所示，Agent 的执行过程就是一个循环。要理解这个循环，可以从最简单的版本开始，逐层升级。

最基础最核心的就是一个条件恒为true的循环，当任务完成后才能退出这个循环。

具体来说就是在这个循环中，调用LLM的接口，传入用户文本，接口返回数据后退出循环，这就是最简单的一个实现了。

如果在进入这个循环前，定义一些tools，然后进入循环之后，传入用户文本的同时也传入tools列表，那么LLM就可以使用工具了。

如果在这个过程中，将用户文本，tools执行结果，以及LLM的最终结论分析记录到数组里，在下次任务时传入LLM，那么就让LLM有了更丰富的上下文了。

以归因查询为例，messages 数组在每轮循环中是这样增长的：

```
第 1 轮: [user: "查一下用户123的归因数据"]
第 2 轮: [user: "查一下用户123的归因数据", assistant: {调用 getUser}, tool: "{status: active, uin: 123}"]
第 3 轮: [..., assistant: {调用 getClick}, tool: "{click_time: 2026-07-15, ad_id: 456}"]
第 4 轮: [..., assistant: {调用 getConfig}, tool: "{attribution_window: 7d}"]
第 5 轮: [..., assistant: "用户123 归因结果：广告ID 456，归因窗口内最后点击"]
```

可以看到，每一轮都在上一次的基础上追加新信息，LLM 的上下文越来越丰富，最终不再调用工具，直接输出结论。

至此就完成了从纯思考的Agent，升级为了可思考可实施且有上下文的Agent。

## 5. 怎么保证不崩

至此Agent已经能run了，但是可能会有人问，「听着挺好，但万一它乱来怎么办？」

是的，LLM是个概率模型，它并不会保证每次执行的输出都是一致的。

目前掌握的就是

1. 保证LLM的输出json一定是符合schema标准的，可以有3层保证
    a. 较弱，通过prompt，强调要使用json格式，保证95%。
    b. 通过OpenAI的strict mode，在LLM内部通过数学方式保证。
    c. 在工具中做容错处理，避免整个工具崩掉，只是这一步异常。
2. 禁止对工具返回结果做LLM总结、分析，因为每次LLM处理都有出问题的概率，通过这步避免对实时数据产生污染。
