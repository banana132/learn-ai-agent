## M1-W1-1 · ReAct 是什么 · v1 卡 (2026-07-14 生成)

### 卡片 1: ReAct 三动作

**Q**: ReAct 三个动作是什么? 用一句话说清关系。

**A**: Reason(想) → Act(用工具) → Observe(看结果) → 循环
核心: 每一轮都在 messages 数组末尾追加新消息,直到 LLM 不再返回 tool_calls。

---

### 卡片 2: ReAct vs CoT

**Q**: ReAct 和 CoT (Chain of Thought) 的核心区别?

**A**: CoT 只想不动手(单次输出思考步骤)。
ReAct = CoT + 工具调用 + 外部反馈闭环 + 循环。
没工具的 ReAct 退化成 CoT。

---

### 卡片 3: 运维类比

**Q**: 用你熟悉的运维场景解释 ReAct。

**A**: LLM 排查告警的过程:
看告警 → kubectl logs → 看日志 → kubectl top → 看内存 → 结论
ReAct 就是把这个流程用 3 段 prompt 教给 LLM。

### 卡片 4: 三元组的工程语义 ⭐

**Q**: ReAct 三元组从【工程视角】看, 各是什么? Observation 和 Thought 最容易混在哪?

**A**:
- **Thought** = LLM 【主动推理输出】(决定下一步做什么)
- **Action** = LLM 输出的【结构化工具调用】(tool_call JSON)
- **Observation** = 工具执行的【原始返回值】, 【原样】塞回 context

⚠️ 易混点: 拿到工具返回后,【不要】让 LLM 先 interpret 再塞 context。
observation = tool_result, 就这么简单, 一字不改。

工程类比: 微服务里 orchestrator 收到 service 返回值, 直接进下一步; 不会先找"解读员"翻译。

---

### 卡片 5: 为什么 Observation 不能加工? (3 大死因 + 深层原因) ⭐⭐

**Q**: 为什么 ReAct 循环中 Observation 必须原样塞 context, 不能让 LLM 先 interpret?

**A**: 3 个致命理由:

1. **信息损失**: JSON 里的字段被总结掉
   例: `{"price": 245.3, "volume": 1M, "trend": "up"}` → "股价上涨" → volume 永久丢失
2. **幻觉入口**: 每轮多 1 次 LLM 调用 = 多 1 次瞎编机会
3. **成本翻倍**: 4 轮循环从 4 次 LLM 调用变 8 次

深层原因: 破坏 **Faithfulness (忠实性)**
→ 后续推理建在失真数据上
→ 错误【指数级放大】(agent 幻觉复合)

轮次: 1→2→3→4, 失真: 1%→3%→10%→30% → agent 彻底跑偏

---

### 卡片 6: Type Drift 灾难 (我自己举的例子) ⭐

**Q**: 举一个"Observation 被加工"导致的真实 bug 类型?

**A**: **Type Drift (类型漂移)**

例 1 (广告归因):
- 数据库 uin: `"00123"` (字符串)
- LLM 加工: `123` (int)
- 下一步 SQL: `WHERE uin = 123` → 💥 查不到

例 2 (退款 API):
- API 返回: `{"user_id": "007", "amount": "1.00"}`
- LLM 总结: "user 7 花了 1 块钱"
- 下一轮调用: `refund(user_id=7, amount=1)` → 💥 退错人!

铁律: 让 LLM 读原始数据, 不要预处理。LLM 就是干这个用的。

---

### 卡片 7: ReAct 一句话本质 + 相比 CoT 的信息论优势 ⭐

**Q**: ReAct 相比纯 CoT 的【本质差异】是什么? 用信息论视角。

**A**:
- CoT 的推理原料 = 训练时刻的【旧知识快照】 (静态, 会瞎编)
- ReAct 的推理原料 = 每轮 tool 返回的【实时数据】 (动态, 复利叠加)

心智模型: **复利**
轮次 1→2→3→4, context 信息量 10%→30%→70%→95%

关键辨别: 没有外部工具的 ReAct 退化成 CoT。
"ReAct 就是 LLM 循环调自己" ← ❌ 常见误解
真相: ReAct 是 LLM + 外部世界的循环对话。
