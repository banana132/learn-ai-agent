## W1: 场景建模 (2026-07-20)

### 1. Agent 角色

**选 B：诊断建议者**

Agent 替人工做信息收集、定位问题，最终决策（问题确认、代码 review、发布）由人工审核批准。

### 2. 核心场景

**场景：归因数据离线与实时不一致告警**

```
收到告警 → Agent 查实时表和离线表 → 定位有差异的用户 ID
  → 查日志 + 配置 → 分析差异原因（没收到数据 / 逻辑过滤 / 其他）
  → 输出诊断结论
  → 人基于输出判断是否是问题，决定是否修改代码
```

### 3. 绝不做的三件事

- ❌ 不经确认重启服务
- ❌ 不经确认更新线上版本
- ❌ 不经确认做线上数据库修改

### 4. 成功标准

| 维度 | 指标 | 目标值 |
|------|------|--------|
| 效率 | 排查时间 | 从半天缩短到 ≤30 分钟 |
| 准确率 | 根因诊断被人工确认正确 | ≥ 80% |

## W2: 工具设计 (2026-07-20)

### 1. 工具清单

| 工具名 | 功能 | 输入 | 输出 | 权限 |
|--------|------|------|------|------|
| `query_logs` | 查线上服务日志 | keyword, time_range | JSON (匹配日志行) | 只读 |
| `query_realtime_db` | 查实时归因数据 | sql (只读) | JSON (查询结果) | 只读 |
| `query_offline_db` | 查离线归因数据 | sql (只读) | JSON (查询结果) | 只读 |
| `get_config` | 读取线上服务配置 | service_name | JSON (配置内容) | 只读 |
| `search_incidents` | 查历史相似故障 | description, limit | JSON (匹配故障列表) | 只读 |
| `search_code` | 查本地代码仓库 | keyword, file_pattern | JSON (匹配代码片段) | 只读（仅本地） |
| `update_config` | 修改线上配置 | service_name, key, value | JSON (变更结果) | ⚠️ 写·需确认 |

### 2. 设计决策

**为什么保留 search_code？**

> 没有查代码工具 → 无法将线上日志跟具体代码逻辑关联 → 诊断链断裂。只查本地代码仓库，不触碰生产环境。

**权限模型**

```
工具内部自治权限检查，不改 Registry 代码:

  Registry.dispatch()     → 不改，只负责查找+调用（开闭原则）
                           ↓
  query_logs()            → 只读，直接执行
  search_code()           → 只读 + 仅限本地路径
  update_config()         → 写工具 → 弹确认 → 等待人批准 → 执行
```

### 3. 最危险的工具

`update_config` — 配置修改是最常见的运维写操作，但错误配置可能引发连锁故障。

安全门禁：
- ① 人工确认弹窗（必须）
- ② 变更前自动备份当前配置
- ③ 单次只改一个 key，禁止批量修改

## W3: 架构选型 (2026-07-20)

### 1. 选型结论

**单 Agent。** 归因不一致诊断链每一步依赖前一步结果，天然串行——单 Agent 的 while loop 就是为串行决策链设计的。

### 2. 诊断流程

```
收到告警: "归因实时离线不一致"
            │
    ┌───────┴───────┐
    │               │
查实时表          查离线表        ← 并行（单 Agent 内多 tool_call）
    │               │
    └───────┬───────┘
            │
     LLM 分析差异用户 ID
            │
     查日志(user_ids)
            │
     查代码(error_pattern)
            │
     LLM 生成诊断结论
            │
     输出给人
```

> 注：前两步查询互不依赖，可在单 Agent 的一次 LLM 调用中同时返回多个 tool_call 实现并行。

### 3. 升级信号

当出现以下**可观测信号**时，升级到 Orchestrator：

| 信号 | 触发条件 | 观测方式 |
|------|---------|---------|
| 独立子系统数超标 | 单次诊断中 ≥3 个互相独立的查询 | M3 trajectory 埋点统计 |
| 工具集分化 | 不同子系统需要完全不同的工具集 + 专业 prompt | 工具调用分布分析 |

只要没触发这两个信号，单 Agent 就是最优解——不要为了"架构好看"提前升级。

## W4: 可靠性工程 (2026-07-20)

### 1. 四层防线

| 层 | 机制 | 位置 | 来源 |
|----|------|------|------|
| L1 | 重试+指数退避 | `_chat_with_retry()` | M2 已有 |
| L2 | 工具超时 | `dispatch()` 内部 | M4 新增 |
| L3 | 诊断超时 | while 循环外层 | M4 新增 |
| L4 | 降级输出 + 人工接管 | dispatch 回填 + 最终输出 | M4 新增 |

### 2. 超时策略

| 超时类型 | 数值 | 理由 |
|---------|------|------|
| 单次工具调用 | 30s | DB 查询、日志搜索——超过 30s 说明数据量异常或服务挂了，不等 |
| 整次诊断 | 5min | 目标从半天→半小时，5min 是安全上限——超时交出半成品，不让人干等 |

### 3. 降级优先级

工具挂了不崩，降级继续：

| 优先级 | 工具 | 挂了怎么降级 |
|--------|------|-------------|
| 🔴 致命 | query_realtime_db, query_offline_db | 核心数据缺失，无法诊断→人工接管 |
| 🟡 重要 | query_logs, get_config, search_code | 标注缺失，用剩余工具继续 |
| 🟢 可降级 | search_incidents | 缺少历史参考不影响本次诊断 |

### 4. 人工接管协议

```
## Agent 诊断报告 (诊断耗时 3min)

✅ 已查明:
  - 离线表用户 12345 有归因记录，实时表无 → 差异确认

❌ 未完成:
  - query_logs 超时，日志数据未获取

🔮 推测:
  - 根据已有信息，可能是实时链路 Kafka 消费延迟导致数据延迟写入

⚠️ 待确认:
  - 查 Kafka 消费 lag 后确认根因

📊 置信度: 中（缺少日志验证，推测基于数据对比，非直接证据）
```

> 目标：人类 30 秒内能接上诊断，不是从头开始。
